Topic 1

Foundations of Systems Administration

This lecture establishes the basic language of systems administration before students move into AWS infrastructure, Linux administration, and operational maintenance.

Lecture focus

  • Understand what systems administration is.
  • Connect infrastructure, operating systems, and cloud platforms.
  • Explain why AWS changes, but does not remove, administrative responsibility.

Subtopics

  • Systems Theory
  • Computer Systems and IT Infrastructure
  • Operating Systems Fundamentals
  • Client-Server Architecture
  • Roles and Responsibilities of Systems Administrators
  • AWS Global Infrastructure
  • AWS Shared Responsibility Model

Objectives

What students should be able to explain

  • Explain the principles, functions, and responsibilities of systems administration based on systems theory and operating system concepts.
  • Differentiate traditional on-premises infrastructure from cloud-based infrastructure using AWS Global Infrastructure.
  • Describe how AWS Regions and Availability Zones support distributed and resilient computing.
  • Explain the AWS Shared Responsibility Model and its implications for cloud administration.

Source framing: NIST SP 800-145, NIST SP 800-144, AWS Regions and Availability Zones documentation.

Chapter insight

What exactly does a systems administrator do?

Core idea

System administration is an operational discipline concerned with keeping computer systems useful, available, secure, and maintainable over time. The role changes by organization, but the underlying responsibility is always service continuity.

Academic takeaways

  • Administration includes provisioning, configuration, maintenance, troubleshooting, and policy enforcement.
  • Good administration combines human judgment with increasing levels of automation.
  • Documentation and repeatable procedures are part of the technical work, not an optional add-on.
  • The unit of concern is the service delivered to users, not just the machine itself.

Based on Jan Schaumann, An Introduction to System Administration, Ch. 1, and Mark Burgess, On the theory of system administration.

Subtopic 1

Systems Theory

Main idea

A system is best understood as an organized set of interdependent components whose behavior must be understood in relation to goals, feedback, and environmental change. For systems administrators, this means avoiding isolated troubleshooting when the problem is systemic.

Why it matters in administration

  • Inputs, processes, outputs, and feedback loops help explain operational behavior.
  • One component failure can propagate through dependencies such as DNS, identity, storage, or network routing.
  • Administrative decisions must be evaluated at the system level: performance, security, and reliability are linked.

Simple systems view

Input User requests
→
Process System services
→
Output Delivered service
↺
Feedback Logs, alerts, user reports

Conceptual grounding: general systems theory and Burgess’s argument that administration needs a theoretical model linking policy, automation, and outcomes.

Subtopic 2

Computer Systems and IT Infrastructure

Core components

  • Compute: processors, memory, and execution capacity
  • Storage: persistent data and recovery mechanisms
  • Networking: communication paths and service reachability
  • Identity and access: who can do what, and where
  • Applications and services: the business-facing workloads users actually consume

Infrastructure perspective

Infrastructure is not just hardware. In modern administration it includes the operational platform, control plane, service relationships, and the procedures that keep systems running under real constraints such as downtime, risk, and change.

Course framing supported by Schaumann’s introduction to the field and NIST cloud-computing language about configurable resources.

Subtopic 3

Operating Systems Fundamentals

What an OS does

  • Virtualizes hardware resources for programs.
  • Schedules execution and coordinates concurrent processes.
  • Manages memory and persistent storage.
  • Provides interfaces, permissions, and abstractions for users and applications.

Academic view

OSTEP frames operating systems around virtualization, concurrency, and persistence. That view is useful for administrators because many practical failures are really failures of scheduling, resource isolation, storage behavior, or access control.

Source: Arpaci-Dusseau and Arpaci-Dusseau, Operating Systems: Three Easy Pieces.

Subtopic 4

Client-Server Architecture

Basic model

  • Clients initiate requests for data or services.
  • Servers provide shared services, state, or processing capacity.
  • The network is not a side detail; it is part of the architecture and affects latency, reliability, and security.

Administrative importance

Administrators must know request flow, service boundaries, failure domains, and trust boundaries. A user-visible outage often comes from a broken dependency in the server path rather than from the client itself.

Concept supported by standard software-architecture treatments of client/server systems and service-boundary design.

Subtopic 5

Roles and Responsibilities of Systems Administrators

Typical responsibilities

  • Provision systems and maintain operational baselines.
  • Manage users, permissions, configuration, and change.
  • Monitor health, detect anomalies, and troubleshoot failures.
  • Protect systems through patching, access control, and recovery planning.
  • Document architecture, procedures, and incident knowledge for continuity.

Professional view

Administration is both technical and organizational. The objective is not simply to build systems, but to sustain service quality while balancing automation, policy, risk, and human operational work.

Source emphasis: Schaumann on breadth of responsibility; Burgess on balancing automation and intelligent intervention.

Chapter insight

System administration as a profession and field of study

Important points

  • The field is real and demanding, even if job titles vary and work is often under-recognized.
  • Experience matters, but unstructured experience alone does not produce good judgment.
  • Formal models, best practices, and theory improve the quality of operational decisions.
  • Students need guided hands-on work so that theory becomes usable practice.

Why this matters in SAM10

SAM10 should not reduce administration to memorized commands. It should build principled reasoning, operational discipline, and the ability to justify why one configuration or action is better than another.

Based on Schaumann’s professional framing of the discipline and Burgess’s argument for theory-informed administration.

Subtopic 6

AWS Global Infrastructure

Key terms

  • Region: a separate geographic area.
  • Availability Zone: an isolated location within a Region.
  • Local and edge facilities: placements closer to users for latency-sensitive workloads.

Why it matters operationally

AWS designs Regions to be isolated from one another for fault tolerance, and Availability Zones to be independent within a Region. This lets administrators choose where resources live and how much failure isolation a workload needs.

Region and Availability Zones

Region One geographic area containing multiple isolated Availability Zones
AZ 1 Workloads can run here
AZ 2 Workloads can fail over here

Source: AWS Regions and Availability Zones documentation and AWS fault-isolation guidance.

Subtopic 7

AWS Shared Responsibility Model

AWS is responsible for

  • Security of the cloud: facilities, hardware, networking, and foundational platform layers.
  • Availability and resilience of the provider-operated infrastructure.
  • Managed-service controls up to the boundary defined by the service model.

Customer is responsible for

  • Security in the cloud: identities, data handling, workload configuration, and access control.
  • Operating systems, applications, patching, and backups where the selected service model requires it.
  • Configuration quality: a secure provider does not make insecure customer settings safe.

Responsibility split

SAM10 on AWS side Facilities, hardware, and core cloud platform protection
Customer side Accounts, data, OS settings, and workload configuration

Source: AWS shared-responsibility guidance and NIST SP 800-144 on cloud security and outsourcing considerations.

Recap

Key takeaways and short check

Recap

  • Systems administration is a theory-informed operational discipline, not only a collection of commands.
  • Operating systems, service dependencies, and architecture boundaries remain central in cloud environments.
  • AWS changes the delivery model, but it does not remove the customer’s responsibility for secure and correct administration.

Short check

  1. What is one example of feedback in a system?
  2. Why do Regions and Availability Zones matter?
  3. Give one responsibility that stays with the customer in AWS.
