Topic 1

Foundations of Systems Administration Lab

Build and analyze a basic cloud-based system using core infrastructure and AWS concepts.

3 hours

Context

Scenario and scope

Core framing details for the activity before students begin the hands-on work.

Dummy case study

A small training company is moving its file server and internal web portal from a single office server room into AWS. The administrator must explain the basic infrastructure, identify responsibilities, and document the move clearly for management and support staff.

Title

Foundations of Systems Administration Laboratory Activity

Description

Students examine a small infrastructure migration scenario and document the foundational concepts, responsibilities, and cloud relationships that a systems administrator must understand before moving services to AWS.

Learning objectives

  • Identify the major elements of a basic IT environment.
  • Differentiate on-premises and AWS-based infrastructure.
  • Explain AWS global infrastructure and the Shared Responsibility Model.
  • Produce clear infrastructure documentation for operational use.

Execution

Schedule and learning flow

The time-boxed lab sequence and the main teacher and student activity blocks.

3-hour structure

  • Meeting format: 1 session of 3 hours or 2 meetings of 1.5 hours each.
  • 0:00-0:30 review the case study and key concepts.
  • 0:30-1:30 build the on-premises and AWS infrastructure diagrams.
  • 1:30-2:15 complete the comparison and shared-responsibility worksheet.
  • 2:15-3:00 finalize the written artifact and short reflection.

Teacher activities

  • Introduce the case study and review systems, cloud, and shared-responsibility concepts.
  • Show examples of infrastructure diagrams and operational documentation.
  • Guide students through identifying AWS and customer responsibilities.
  • Facilitate discussion on how administration changes in cloud environments.

Student learning activities

  1. Identify and label at least 6 components in a simple on-premises environment: client, server, operating system, network, storage, administrator.
  2. Redraw the same environment in AWS terms and include at least 1 Region and 2 Availability Zones.
  3. Write a comparison table that explains at least 4 differences between traditional infrastructure and AWS cloud infrastructure.
  4. Classify at least 6 responsibilities under either AWS or customer responsibility, with at least 5 correct classifications required for completion.
  5. Use the AWS console demo or AWS documentation to identify where Region and service navigation appear in the interface.
  6. Write a reflection explaining how the systems administrator role changes when infrastructure is moved to AWS.

Linux commands to use and master

  • pwd, ls -la, cd - navigate and inspect the workspace
  • mkdir, cp, mv - organize draft documentation files
  • cat, less, grep - read and review notes and exported text
  • ssh - understand remote administration context even if only simulated in class

Deliverables

Outputs and evaluation

What students submit and how the activity is assessed.

Required outputs

  • One labeled on-premises infrastructure diagram
  • One labeled AWS-based infrastructure diagram
  • One comparison table with at least 4 differences between on-premises and cloud infrastructure
  • One completed shared responsibility worksheet with at least 6 items classified correctly
  • One reflection paragraph of 120 to 180 words

Written artifact to produce

  • Create an `infrastructure overview document` for the dummy company.
  • It must include system purpose, major components, ownership, cloud versus on-premises notes, and shared responsibility notes.
  • The document should be written as if it will be handed to another systems administrator.

Assessment methodologies

  • Diagram rubric for completeness and correct terminology.
  • Worksheet check for shared-responsibility accuracy.
  • Document review for clarity, structure, and operational usefulness.
  • Short written reflection assessed for conceptual understanding.

Skills reinforced

  • Systems thinking
  • Client-server analysis
  • Infrastructure comparison
  • AWS foundational literacy
  • Administrative responsibility boundaries

Support

Guidance and references

Support notes and reference material students can use while completing the lab.

Information for students

  • Use simple diagrams; accuracy matters more than artistic detail.
  • When comparing on-premises and cloud systems, focus on ownership, location, scaling, and resilience.
  • In the Shared Responsibility Model, AWS secures the infrastructure of the cloud, while customers secure their workloads and configurations in the cloud.

AI use note

AI support may be used if needed, but students must follow ethical AI policies for education, verify outputs, and submit work they can clearly explain.

Useful resources

  • AWS Global Infrastructure
  • AWS Regions and Availability Zones documentation
  • EC2 guide to Regions and Availability Zones
  • AWS Shared Responsibility Model

Video lectures and internet materials

  • AWS Cloud Essentials - includes introductory video material on cloud concepts and global infrastructure.
  • AWS Global Infrastructure page - includes AWS-produced explainer media on Regions and Availability Zones.
  • AWS Cloud Practitioner Essentials course overview - points students to current beginner AWS learning paths.

Topic 2

Cloud Computing and Virtualization Fundamentals Lab

Compare cloud deployment models, inspect virtualization approaches, and estimate basic AWS compute costs.

3 hours

Context

Scenario and scope

The scenario, purpose, and learning targets for the lab.

Dummy case study

A startup must choose between on-premises virtualization, AWS EC2 instances, and a container-based deployment for a new internal business application. The team needs a short recommendation and cost-aware deployment comparison.

Title

Cloud Computing and Virtualization Fundamentals Laboratory Activity

Description

Students evaluate deployment and virtualization options for a startup and prepare a short recommendation that balances cloud models, service models, virtualization, and cost.

Learning objectives

  • Compare cloud deployment and service models.
  • Differentiate hypervisors, virtual machines, and containers.
  • Relate AWS compute services to infrastructure decisions.
  • Document a cost-aware deployment recommendation.

Execution

Schedule and learning flow

The time plan and the main activity sequence for the lab.

3-hour structure

  • Meeting format: 1 session of 3 hours or 2 meetings of 1.5 hours each.
  • 0:00-0:30 review deployment models, virtualization, and the startup scenario.
  • 0:30-1:20 complete cloud-model and virtualization comparisons.
  • 1:20-2:10 map AWS services and estimate basic pricing.
  • 2:10-3:00 write the recommendation memo and submit outputs.

Teacher activities

  • Present the startup scenario and review cloud and virtualization models.
  • Demonstrate how to compare service models and pricing assumptions.
  • Guide students through framing a deployment recommendation.
  • Support discussion on trade-offs between flexibility, cost, and manageability.

Student learning activities

  1. Classify at least 4 scenarios into public, private, hybrid, or multi-cloud and justify each answer in one sentence.
  2. Compare virtual machines and containers using at least 5 criteria such as isolation, startup speed, resource use, operating system dependency, and management overhead.
  3. Explain the role of a hypervisor and identify whether it supports virtualization directly or through a host operating system.
  4. Match at least 5 AWS services or use cases to IaaS, PaaS, or SaaS-style delivery models.
  5. Use AWS Pricing Calculator or EC2 pricing pages to estimate a small deployment and record the assumptions used.
  6. Submit a recommendation that balances technical fit, flexibility, and cost.

Linux commands to use and master

  • free -h, top, ps aux - connect compute choices to resource usage
  • uname -a, hostnamectl - inspect host characteristics
  • grep, awk, sort - organize comparison notes and exported values
  • tar, gzip - package lab outputs and supporting files

Deliverables

Outputs and evaluation

Submission requirements and assessment checkpoints.

Required outputs

  • One completed deployment model classification worksheet
  • One virtualization comparison table covering hypervisors, virtual machines, and containers
  • One AWS service-model mapping exercise covering IaaS, PaaS, and SaaS examples
  • One short pricing estimate or cost-planning summary using AWS pricing resources
  • One recommendation paragraph explaining the best deployment choice for the scenario

Written artifact to produce

  • Create a `deployment recommendation memo` for the startup.
  • It must include chosen deployment model, virtualization approach, estimated trade-offs, and a short cost-planning note.
  • The memo should be concise enough for a technical manager to review quickly.

Assessment methodologies

  • Classification worksheet marking guide.
  • Comparison rubric for virtualization analysis quality.
  • Pricing estimate review for assumptions and correctness.
  • Memo assessment for technical justification and communication clarity.

Skills reinforced

  • Cloud model analysis
  • Virtualization literacy
  • Service model differentiation
  • AWS compute awareness
  • Cost-conscious infrastructure planning

Support

Guidance and references

Support notes and external materials that help students complete the activity.

Information for students

  • Focus on why a deployment model fits a scenario, not only on definitions.
  • Containers and virtual machines solve different problems; do not assume one always replaces the other.
  • Cost estimates are only meaningful when the assumptions are explicit.

AI use note

AI support may be used if needed, but students must follow ethical AI policies for education, verify outputs, and submit work they can clearly explain.

Useful resources

  • AWS Cloud Essentials
  • Amazon EC2 pricing
  • AWS Pricing Calculator documentation
  • Generate an estimate in AWS Pricing Calculator
  • Generating Amazon EC2 estimates

Video lectures and internet materials

  • AWS Cloud Essentials - beginner learning materials covering cloud models and core AWS concepts.
  • AWS Cloud Practitioner Essentials course overview - current entry point for official AWS foundational training.
  • AWS Pricing Calculator documentation - useful alongside lecture or demo walkthroughs for cost-planning exercises.

Topic 3

Identity, Access Management, and Governance Lab

Design secure access controls, evaluate IAM policies, and apply governance principles to AWS administration scenarios.

3 hours

Dummy case study

A school IT team has multiple staff roles: registrar, finance officer, faculty support, and cloud administrator. Recent access complaints and over-permissioned accounts require a documented IAM and governance cleanup plan.

Title

Identity, Access Management, and Governance Laboratory Activity

Description

Students redesign a school IT team’s access model and produce a draft governance document that aligns IAM decisions with secure administration practices.

3-hour structure

  • Meeting format: 1 session of 3 hours or 2 meetings of 1.5 hours each.
  • 0:00-0:30 review authentication, authorization, IAM, and the case study.
  • 0:30-1:20 compare RBAC and ABAC, then assign role permissions.
  • 1:20-2:10 evaluate MFA, federation, and least-privilege controls.
  • 2:10-3:00 complete the governance note and checklist.

Learning objectives

  • Explain authentication and authorization in administration contexts.
  • Differentiate RBAC and ABAC.
  • Apply least-privilege thinking to IAM design.
  • Produce a governance-oriented access control document.

Required outputs

  • One access-control comparison sheet covering RBAC and ABAC
  • One IAM role or policy design recommendation for a given scenario
  • One least-privilege review of an identity policy set
  • One governance checklist with at least 5 secure-administration controls
  • One short analysis paragraph on MFA and identity federation

Written artifact to produce

  • Create an `access control and governance policy note` for the school IT team.
  • It must include roles, permission boundaries, MFA expectations, review rules, and least-privilege justification.
  • The document should read like an internal policy draft.

Teacher activities

  • Introduce the school access-control scenario and clarify role boundaries.
  • Review IAM, MFA, federation, and least-privilege concepts.
  • Model how governance requirements become written policy.
  • Facilitate peer critique of over-permissioned access designs.

Student learning activities

  1. Define authentication and authorization in your own words and give one administration example of each.
  2. Compare RBAC and ABAC using at least 4 criteria such as flexibility, complexity, scalability, and policy management.
  3. Review a sample AWS administration scenario and assign permissions using least-privilege logic.
  4. Identify at least 3 cases where MFA should be required for secure administrative access.
  5. Explain how identity federation reduces the need for separate long-term IAM user accounts.
  6. Create a short governance checklist covering account review, policy review, MFA enforcement, role use, and access monitoring.

Linux commands to use and master

  • id, groups, whoami - understand identity context on a Linux system
  • chmod, chown, chgrp - connect permissions to least-privilege practice
  • sudo - understand elevated execution and privilege separation
  • grep, cat, less - inspect mock policy and access files

Assessment methodologies

  • Scenario-based marking for least-privilege decisions.
  • Comparison rubric for RBAC and ABAC analysis.
  • Policy note review for governance coverage and practical clarity.
  • Short written checks on MFA and federation understanding.

Skills reinforced

  • Identity and access analysis
  • Least-privilege reasoning
  • IAM policy evaluation
  • Governance awareness
  • Secure cloud administration planning

Information for students

  • Do not treat IAM as only a permissions list; it is part of a larger governance model.
  • Least privilege means enough access to perform the task, not the fewest words in a policy.
  • Federation is useful when organizations already manage identities in an external directory or identity provider.

AI use note

AI support may be used if needed, but students must follow ethical AI policies for education, verify outputs, and submit work they can clearly explain.

Useful resources

  • AWS Identity and Access Management documentation
  • Policies and permissions in AWS IAM
  • Multi-factor authentication in IAM
  • AWS identity federation
  • Security best practices in IAM
  • AWS Well-Architected least privilege guidance

Video lectures and internet materials

  • AWS Cloud Essentials - current foundational learning materials that include IAM and security basics.
  • AWS Cloud Practitioner Essentials course overview - entry point to official foundational AWS training.
  • AWS identity federation overview - public overview material useful before deeper configuration work.

Topic 4

Computer Networks and Cloud Networking Lab

Model secure VPC networking by designing subnets, route tables, gateways, and traffic controls for an AWS-based environment.

3 hours

Dummy case study

A company is deploying a web application in AWS with a public-facing frontend and a private backend database. The administrator must design the VPC layout, traffic flow, and control points before deployment.

Title

Computer Networks and Cloud Networking Laboratory Activity

Description

Students design a secure AWS network for a web application and produce a short network design document that explains subnetting, routing, gateways, and traffic control.

3-hour structure

  • Meeting format: 1 session of 3 hours or 2 meetings of 1.5 hours each.
  • 0:00-0:30 review OSI, TCP/IP, VPC, and the web-application case.
  • 0:30-1:25 create the VPC diagram, subnet layout, and route plan.
  • 1:25-2:10 compare security groups and network ACLs and explain DNS use.
  • 2:10-3:00 complete the network design brief and final evaluation.

Learning objectives

  • Explain the role of routing, subnets, DNS, and gateways.
  • Relate OSI and TCP/IP models to cloud networking practice.
  • Analyze VPC components and security controls.
  • Document a basic secure cloud network design.

Required outputs

  • One OSI versus TCP/IP comparison table
  • One VPC network diagram with public and private subnets
  • One route-table plan that shows internet and private outbound traffic flow
  • One security-control comparison sheet for security groups and network ACLs
  • One short evaluation paragraph explaining whether the design is secure and functional

Written artifact to produce

  • Create a `network design brief` for the company web application.
  • It must include VPC components, subnet purposes, route intent, gateway roles, and traffic security notes.
  • The document should be clear enough for another administrator to implement the design.

Teacher activities

  • Introduce the network case study and review VPC architecture concepts.
  • Demonstrate how route tables and gateways affect public and private connectivity.
  • Compare security groups and network ACLs using example rule sets.
  • Support students in documenting their design rationale clearly.

Student learning activities

  1. Differentiate the OSI and TCP/IP models by matching at least 5 networking functions to the correct model layers.
  2. Create a VPC design that includes at least 1 VPC, 2 subnets, 1 route table, 1 internet gateway, and 1 NAT gateway.
  3. Explain which subnet is public and which is private based on the route-table targets.
  4. Describe how DNS supports access to cloud resources and why routing is required for traffic movement.
  5. Compare security groups and network ACLs using at least 4 criteria such as scope, statefulness, and rule direction.
  6. Evaluate whether the design supports secure inbound and outbound communication for a web application scenario.

Linux commands to use and master

  • ip a, ip r - inspect addressing and routing
  • ping, traceroute or tracepath - test connectivity paths
  • ss -tuln or netstat -tuln - inspect listening ports
  • dig, nslookup, host - inspect DNS behavior
  • curl - test service reachability

Assessment methodologies

  • Diagram and route-table rubric for architecture accuracy.
  • Short-answer assessment for DNS, routing, and gateway understanding.
  • Security comparison check for SG and NACL accuracy.
  • Document review for implementation clarity and technical reasoning.

Skills reinforced

  • Cloud networking analysis
  • VPC design literacy
  • Routing and subnet reasoning
  • Traffic-security evaluation
  • Network architecture communication

Information for students

  • A subnet is considered public only when its route table allows traffic to an internet gateway.
  • A NAT gateway supports outbound internet access from private subnets without allowing unsolicited inbound traffic.
  • Security groups and network ACLs both matter because they protect at different scopes.

AI use note

AI support may be used if needed, but students must follow ethical AI policies for education, verify outputs, and submit work they can clearly explain.

Useful resources

  • How Amazon VPC works
  • Subnet route tables in Amazon VPC
  • Configure route tables
  • Internet gateways in Amazon VPC
  • NAT gateways
  • Security groups for your VPC
  • Network ACLs

Video lectures and internet materials

  • Updated AWS Cloud Practitioner Essentials overview - highlights the current AWS networking module in the beginner course.
  • AWS Cloud Practitioner Essentials digital course overview - foundational training path for AWS networking basics.
  • Free AWS training webinars - public AWS training material that can supplement network and cloud lessons.

Topic 5

Compute Infrastructure Administration Lab

Provision and evaluate EC2-based compute environments using AMIs, instance sizing, launch templates, Elastic IP, and basic CloudFormation thinking.

3 hours

Dummy case study

An online training platform needs a new Linux-based EC2 deployment for its staging environment. The administrator must choose compute resources, explain the AMI and launch settings, and document how the environment could later be reproduced with Infrastructure as Code.

Title

Compute Infrastructure Administration Laboratory Activity

Description

Students plan a Linux-based staging deployment on Amazon EC2, justify compute choices, and produce a repeatable provisioning runbook with an introductory Infrastructure as Code perspective.

3-hour structure

  • Meeting format: 1 session of 3 hours or 2 meetings of 1.5 hours each.
  • 0:00-0:30 review EC2 concepts, AMIs, instance types, and the case study.
  • 0:30-1:20 select compute options and compare VMs, containers, and elastic compute.
  • 1:20-2:10 define launch-template and Elastic IP decisions.
  • 2:10-3:00 complete the runbook and short CloudFormation explanation.

Learning objectives

  • Explain EC2 and virtualization in cloud provisioning.
  • Select compute resources based on workload requirements.
  • Explain the purpose of AMIs, launch templates, and Elastic IPs.
  • Describe the value of Infrastructure as Code using CloudFormation.

Required outputs

  • One EC2 deployment plan with AMI, instance type, storage, and networking choices
  • One comparison sheet covering virtual machines, containers, and elastic compute resources
  • One launch template design summary with reusable parameters
  • One short infrastructure-as-code explanation using a simple CloudFormation scenario
  • One recommendation paragraph justifying the compute design

Written artifact to produce

  • Create a `compute provisioning runbook` for the staging environment.
  • It must include AMI choice, instance type rationale, launch template parameters, IP considerations, and CloudFormation notes.
  • The runbook should be detailed enough for repeatable provisioning.

Teacher activities

  • Introduce the staging-environment case study and review EC2 provisioning concepts.
  • Demonstrate how AMIs, instance types, and launch templates affect deployment outcomes.
  • Explain how CloudFormation supports repeatable provisioning.
  • Guide students in turning design choices into a usable runbook.

Student learning activities

  1. Identify an appropriate AMI and instance type for at least 2 different workload scenarios and explain the reasoning.
  2. Differentiate virtual machines, containers, and elastic compute in terms of provisioning, scaling, and management.
  3. Describe what information should be stored in a launch template and why that improves consistency.
  4. Explain how an Elastic IP differs from a standard dynamic public IP in an operational scenario.
  5. Outline how AWS CloudFormation can provision infrastructure repeatedly from a defined template.
  6. Write a final recommendation that balances workload needs, repeatability, and administrative control.

Linux commands to use and master

  • uname -a, hostnamectl - identify system characteristics
  • df -h, lsblk, free -h - inspect compute and storage characteristics
  • systemctl, journalctl - manage services and review logs
  • ssh, scp, rsync - access and transfer files to remote Linux instances

Assessment methodologies

  • Provisioning plan rubric for technical fit and justification.
  • Short written assessment for EC2, launch template, and Elastic IP understanding.
  • Runbook review for repeatability and clarity.
  • CloudFormation explanation assessed for conceptual correctness.

Skills reinforced

  • Compute resource selection
  • Provisioning analysis
  • Reusable configuration design
  • Infrastructure as Code awareness
  • Cloud administration planning

Information for students

  • An AMI is the machine template used to launch an instance; it does not replace instance sizing decisions.
  • Launch templates improve repeatability by storing launch parameters once and reusing them.
  • CloudFormation is valuable because it turns infrastructure setup into a versionable and repeatable definition.

AI use note

AI support may be used if needed, but students must follow ethical AI policies for education, verify outputs, and submit work they can clearly explain.

Useful resources

  • Amazon EC2 documentation
  • Amazon EC2 documentation overview
  • Amazon EC2 instances
  • Store instance launch parameters in launch templates
  • Create an Amazon EC2 launch template
  • Getting started with AWS CloudFormation

Video lectures and internet materials

  • AWS Cloud Practitioner Essentials digital course overview - includes current beginner coverage of EC2 and foundational provisioning concepts.
  • Amazon EC2 documentation overview - public entry point to guided learning resources and documentation for EC2.
  • CloudFormation getting started guide - practical official introduction to Infrastructure as Code with AWS.

Topic 6

Linux Systems Administration Lab

Administer a cloud-hosted Linux server by inspecting the OS, managing users and permissions, reviewing processes, and comparing SSH with AWS Systems Manager workflows.

3 hours

Dummy case study

A training organization has deployed a Linux-based application server in AWS. The systems administrator must inspect the operating environment, verify user and permission settings, review running processes, and recommend secure remote administration methods for support staff.

Title

Linux Systems Administration Laboratory Activity

Description

Students analyze a cloud-hosted Linux server scenario and produce a concise administration guide covering architecture, shell usage, file systems, users, permissions, processes, packages, and secure remote access options.

3-hour structure

  • Meeting format: 1 session of 3 hours or 2 meetings of 1.5 hours each.
  • 0:00-0:30 review Linux architecture, the shell environment, and the case study.
  • 0:30-1:20 inspect files, users, groups, permissions, and package information.
  • 1:20-2:10 review processes, services, and secure remote administration options.
  • 2:10-3:00 complete the Linux administration guide and final evaluation.

Learning objectives

  • Explain Linux architecture and operational principles.
  • Differentiate file systems, users, groups, permissions, and processes.
  • Analyze Linux administration practices in cloud environments.
  • Compare SSH and AWS Systems Manager for secure remote administration.

Required outputs

  • One Linux system inspection worksheet covering architecture, file system layout, and package details
  • One user, group, and permission review sheet with at least 5 observations
  • One process and service summary identifying at least 4 important running components
  • One SSH versus AWS Systems Manager comparison table
  • One short recommendation paragraph on secure Linux administration strategy

Written artifact to produce

  • Create a Linux administration quick guide for the training organization.
  • It must include shell basics, file-system notes, user and permission guidance, process checks, package-management notes, and remote access recommendations.
  • The guide should be useful to another junior systems administrator.

Teacher activities

  • Introduce the cloud-hosted Linux server scenario and review Linux administration concepts.
  • Demonstrate how to inspect users, permissions, processes, services, and packages from the shell.
  • Explain the security and operational trade-offs between SSH and AWS Systems Manager.
  • Guide students in converting observations into a practical administration guide.

Student learning activities

  1. Inspect Linux architecture and environment details using shell commands and identify key system characteristics.
  2. Review the file-system structure and explain the role of at least 5 important directories or files.
  3. Inspect users, groups, and permissions and record at least 5 relevant administration observations.
  4. Review processes or services and identify which components are important for application availability.
  5. Compare package-management and remote-access approaches, including SSH and AWS Systems Manager.
  6. Write a short strategy note explaining how Linux systems should be administered securely in the cloud.

Linux commands to use and master

  • pwd, ls -la, cd, cat - inspect the shell environment and files
  • whoami, id, groups, chmod, chown - analyze users, groups, and permissions
  • ps aux, top, systemctl, journalctl - review processes and services
  • uname -a, hostnamectl, df -h, lsblk - inspect Linux architecture and file systems
  • ssh - understand secure remote administration context

Assessment methodologies

  • Inspection worksheet check for technical accuracy.
  • User and permissions review rubric for completeness and interpretation.
  • Process and service summary review for operational relevance.
  • Comparison table assessment for SSH versus Systems Manager reasoning.
  • Written guide review for practical clarity and enterprise usefulness.

Skills reinforced

  • Linux shell fluency
  • User and permission analysis
  • Process and service inspection
  • Secure remote administration reasoning
  • Cloud-hosted Linux operations planning

Information for students

  • Focus on what a Linux administrator needs to inspect regularly, not only on memorizing command names.
  • Permissions should be interpreted in terms of administrative risk and operational need.
  • SSH remains important, but managed tools such as AWS Systems Manager can reduce exposure and improve control.

AI use note

AI support may be used if needed, but students must follow ethical AI policies for education, verify outputs, and submit work they can clearly explain.

Useful resources

  • AWS Systems Manager documentation
  • AWS Systems Manager Session Manager
  • Connect to your Linux instance using SSH
  • Ubuntu Server documentation
  • Bash Reference Manual

Video lectures and internet materials

  • AWS Cloud Practitioner Essentials digital course overview - entry point to current AWS foundational training.
  • AWS Session Manager guide - useful for understanding remote administration without relying only on inbound SSH.
  • Ubuntu Server documentation - practical Linux administration reference for file systems, users, packages, and services.

Topic 7

Storage Administration Lab

Evaluate enterprise storage needs by comparing block, object, and file storage, then recommend AWS storage services and lifecycle strategies.

3 hours

Dummy case study

A growing organization needs storage for website assets, database volumes, shared team files, and archived records. The systems administrator must recommend appropriate storage architectures and AWS services while balancing performance, availability, and cost.

Title

Storage Administration Laboratory Activity

Description

Students analyze a business storage scenario and prepare a storage strategy that distinguishes file, block, and object storage, connects RAID and performance concepts to enterprise needs, and maps those needs to AWS storage services.

3-hour structure

  • Meeting format: 1 session of 3 hours or 2 meetings of 1.5 hours each.
  • 0:00-0:30 review storage hierarchy, storage architectures, and the case study.
  • 0:30-1:20 compare block, object, and file storage and review RAID and performance concepts.
  • 1:20-2:10 map organizational requirements to Amazon S3, EBS, EFS, and lifecycle policies.
  • 2:10-3:00 complete the storage strategy document and final recommendation.

Learning objectives

  • Differentiate block, object, and file storage architectures.
  • Explain enterprise storage technologies used in cloud infrastructure.
  • Analyze storage requirements and match them to appropriate solutions.
  • Evaluate storage strategies using performance, availability, and cost.

Required outputs

  • One storage comparison worksheet covering block, object, and file storage
  • One short RAID and storage-performance explanation sheet
  • One AWS storage mapping table covering S3, EBS, EFS, and lifecycle policies
  • One storage requirement analysis for at least 4 business needs
  • One final recommendation paragraph justifying the chosen strategy

Written artifact to produce

  • Create a storage strategy brief for the organization.
  • It must include workload-to-storage mapping, performance and availability notes, lifecycle decisions, and cost-aware reasoning.
  • The brief should be practical enough for management review and technical follow-up.

Teacher activities

  • Introduce the storage case study and review storage architecture principles.
  • Explain the differences between object, block, and file storage with enterprise examples.
  • Demonstrate how RAID, throughput, latency, and lifecycle policies affect storage planning.
  • Guide students in translating storage needs into AWS service choices and a written strategy.

Student learning activities

  1. Compare block, object, and file storage using at least 5 criteria such as access model, performance, persistence, and workload fit.
  2. Explain RAID and storage-performance concepts in relation to enterprise systems.
  3. Analyze at least 4 organizational storage needs and select the best-fit storage model for each.
  4. Map each requirement to AWS services such as Amazon S3, Amazon EBS, or Amazon EFS.
  5. Describe how lifecycle policies affect cost and storage management over time.
  6. Write a short final recommendation that balances performance, availability, and cost.

Linux commands to use and master

  • df -h, du -sh, lsblk - inspect file systems and storage usage
  • mount, umount - understand file-system attachment concepts
  • ls -la, cp, mv - connect storage structure to file operations
  • tar, gzip - relate archival and storage handling to enterprise operations
  • iostat or vmstat - discuss basic storage-performance observation where available

Assessment methodologies

  • Storage comparison worksheet check for conceptual accuracy.
  • Requirement-mapping rubric for workload-to-service alignment.
  • Short written assessment for RAID and performance understanding.
