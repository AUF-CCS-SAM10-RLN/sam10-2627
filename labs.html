Topic 1

Foundations of Systems Administration Lab

Build and analyze a basic cloud-based system using core infrastructure and AWS concepts.

3 hours

Goal

Apply systems administration concepts by identifying system components, mapping client-server relationships, and examining how AWS infrastructure and the Shared Responsibility Model affect administration.

Required outputs

  • One labeled on-premises infrastructure diagram
  • One labeled AWS-based infrastructure diagram
  • One comparison table with at least 4 differences between on-premises and cloud infrastructure
  • One completed shared responsibility worksheet with at least 6 items classified correctly
  • One reflection paragraph of 120 to 180 words

Assessable tasks

  1. Identify and label at least 6 components in a simple on-premises environment: client, server, operating system, network, storage, administrator.
  2. Redraw the same environment in AWS terms and include at least 1 Region and 2 Availability Zones.
  3. Write a comparison table that explains at least 4 differences between traditional infrastructure and AWS cloud infrastructure.
  4. Classify at least 6 responsibilities under either AWS or customer responsibility, with at least 5 correct classifications required for completion.
  5. Use the AWS console demo or AWS documentation to identify where Region and service navigation appear in the interface.
  6. Write a reflection explaining how the systems administrator role changes when infrastructure is moved to AWS.

Success criteria

  • Diagrams are complete, readable, and use correct terms.
  • AWS diagrams correctly show Region and Availability Zone structure.
  • The comparison table explains differences, not just lists keywords.
  • The shared responsibility answers correctly distinguish security of the cloud from security in the cloud.
  • The written reflection connects theory to administrative practice.

Skills reinforced

  • Systems thinking
  • Client-server analysis
  • Infrastructure comparison
  • AWS foundational literacy
  • Administrative responsibility boundaries

Information for students

  • Use simple diagrams; accuracy matters more than artistic detail.
  • When comparing on-premises and cloud systems, focus on ownership, location, scaling, and resilience.
  • In the Shared Responsibility Model, AWS secures the infrastructure of the cloud, while customers secure their workloads and configurations in the cloud.

Useful resources

  • AWS Regions and Availability Zones documentation
  • EC2 guide to Regions and Availability Zones
  • AWS Shared Responsibility Model

Topic 2

Cloud Computing and Virtualization Fundamentals Lab

Compare cloud deployment models, inspect virtualization approaches, and estimate basic AWS compute costs.

3 hours

Goal

Connect distributed systems and virtualization theory to practical infrastructure choices, then evaluate AWS compute and pricing decisions for a small organization.

Required outputs

  • One completed deployment model classification worksheet
  • One virtualization comparison table covering hypervisors, virtual machines, and containers
  • One AWS service-model mapping exercise covering IaaS, PaaS, and SaaS examples
  • One short pricing estimate or cost-planning summary using AWS pricing resources
  • One recommendation paragraph explaining the best deployment choice for the scenario

Assessable tasks

  1. Classify at least 4 scenarios into public, private, hybrid, or multi-cloud and justify each answer in one sentence.
  2. Compare virtual machines and containers using at least 5 criteria such as isolation, startup speed, resource use, operating system dependency, and management overhead.
  3. Explain the role of a hypervisor and identify whether it supports virtualization directly or through a host operating system.
  4. Match at least 5 AWS services or use cases to IaaS, PaaS, or SaaS-style delivery models.
  5. Use AWS Pricing Calculator or EC2 pricing pages to estimate a small deployment and record the assumptions used.
  6. Submit a recommendation that balances technical fit, flexibility, and cost.

Success criteria

  • Deployment model classifications are justified using the scenario details.
  • Virtualization comparisons describe trade-offs rather than isolated definitions.
  • AWS service mappings are logically consistent with IaaS, PaaS, and SaaS concepts.
  • Pricing work includes visible assumptions such as instance type, storage, or duration.
  • The final recommendation shows a clear reasoned choice.

Skills reinforced

  • Cloud model analysis
  • Virtualization literacy
  • Service model differentiation
  • AWS compute awareness
  • Cost-conscious infrastructure planning

Information for students

  • Focus on why a deployment model fits a scenario, not only on definitions.
  • Containers and virtual machines solve different problems; do not assume one always replaces the other.
  • Cost estimates are only meaningful when the assumptions are explicit.

Useful resources

  • AWS Cloud Essentials
  • Amazon EC2 pricing
  • AWS Pricing Calculator documentation
  • Generate an estimate in AWS Pricing Calculator
  • Generating Amazon EC2 estimates

Topic 3

Identity, Access Management, and Governance Lab

Design secure access controls, evaluate IAM policies, and apply governance principles to AWS administration scenarios.

3 hours

Goal

Apply authentication, authorization, IAM, least-privilege, MFA, and federation concepts to realistic cloud administration scenarios.

Required outputs

  • One access-control comparison sheet covering RBAC and ABAC
  • One IAM role or policy design recommendation for a given scenario
  • One least-privilege review of an identity policy set
  • One governance checklist with at least 5 secure-administration controls
  • One short analysis paragraph on MFA and identity federation

Assessable tasks

  1. Define authentication and authorization in your own words and give one administration example of each.
  2. Compare RBAC and ABAC using at least 4 criteria such as flexibility, complexity, scalability, and policy management.
  3. Review a sample AWS administration scenario and assign permissions using least-privilege logic.
  4. Identify at least 3 cases where MFA should be required for secure administrative access.
  5. Explain how identity federation reduces the need for separate long-term IAM user accounts.
  6. Create a short governance checklist covering account review, policy review, MFA enforcement, role use, and access monitoring.

Success criteria

  • Authentication and authorization are correctly distinguished.
  • RBAC and ABAC comparisons show practical differences rather than definitions only.
  • IAM recommendations follow least-privilege and do not over-assign permissions.
  • MFA and federation explanations connect directly to secure cloud administration.
  • The governance checklist is actionable and relevant to AWS account operations.

Skills reinforced

  • Identity and access analysis
  • Least-privilege reasoning
  • IAM policy evaluation
  • Governance awareness
  • Secure cloud administration planning

Information for students

  • Do not treat IAM as only a permissions list; it is part of a larger governance model.
  • Least privilege means enough access to perform the task, not the fewest words in a policy.
  • Federation is useful when organizations already manage identities in an external directory or identity provider.

Topic 4

Computer Networks and Cloud Networking Lab

Model secure VPC networking by designing subnets, route tables, gateways, and traffic controls for an AWS-based environment.

3 hours

Goal

Apply core networking concepts to cloud infrastructure by mapping traffic flow, designing VPC components, and evaluating secure communication paths.

Required outputs

  • One OSI versus TCP/IP comparison table
  • One VPC network diagram with public and private subnets
  • One route-table plan that shows internet and private outbound traffic flow
  • One security-control comparison sheet for security groups and network ACLs
  • One short evaluation paragraph explaining whether the design is secure and functional

Assessable tasks

  1. Differentiate the OSI and TCP/IP models by matching at least 5 networking functions to the correct model layers.
  2. Create a VPC design that includes at least 1 VPC, 2 subnets, 1 route table, 1 internet gateway, and 1 NAT gateway.
  3. Explain which subnet is public and which is private based on the route-table targets.
  4. Describe how DNS supports access to cloud resources and why routing is required for traffic movement.
  5. Compare security groups and network ACLs using at least 4 criteria such as scope, statefulness, and rule direction.
  6. Evaluate whether the design supports secure inbound and outbound communication for a web application scenario.

Success criteria

  • The VPC design correctly distinguishes public and private connectivity paths.
  • Routing decisions align with the selected gateways and subnet intent.
  • DNS, firewalls, routing, and subnets are explained in operational terms, not only definitions.
  • Security groups and network ACLs are compared accurately.
  • The final evaluation clearly judges both functionality and security.

Skills reinforced

  • Cloud networking analysis
  • VPC design literacy
  • Routing and subnet reasoning
  • Traffic-security evaluation
  • Network architecture communication

Information for students

  • A subnet is considered public only when its route table allows traffic to an internet gateway.
  • A NAT gateway supports outbound internet access from private subnets without allowing unsolicited inbound traffic.
  • Security groups and network ACLs both matter because they protect at different scopes.

Topic 5

Compute Infrastructure Administration Lab

Provision and evaluate EC2-based compute environments using AMIs, instance sizing, launch templates, Elastic IP, and basic CloudFormation thinking.

3 hours

Goal

Apply compute-administration concepts by selecting EC2 resources, comparing provisioning methods, and introducing Infrastructure as Code with AWS CloudFormation.

Required outputs

  • One EC2 deployment plan with AMI, instance type, storage, and networking choices
  • One comparison sheet covering virtual machines, containers, and elastic compute resources
  • One launch template design summary with reusable parameters
  • One short infrastructure-as-code explanation using a simple CloudFormation scenario
  • One recommendation paragraph justifying the compute design

Assessable tasks

  1. Identify an appropriate AMI and instance type for at least 2 different workload scenarios and explain the reasoning.
  2. Differentiate virtual machines, containers, and elastic compute in terms of provisioning, scaling, and management.
  3. Describe what information should be stored in a launch template and why that improves consistency.
  4. Explain how an Elastic IP differs from a standard dynamic public IP in an operational scenario.
  5. Outline how AWS CloudFormation can provision infrastructure repeatedly from a defined template.
  6. Write a final recommendation that balances workload needs, repeatability, and administrative control.

Success criteria

  • Compute choices are justified using workload characteristics rather than guesswork.
  • AMI, instance type, and launch template roles are clearly distinguished.
  • Elastic IP use is explained in a realistic administration context.
  • Infrastructure as Code is described as a provisioning method, not only a definition.
  • The final recommendation is operationally coherent and technically defensible.

Skills reinforced

  • Compute resource selection
  • Provisioning analysis
  • Reusable configuration design
  • Infrastructure as Code awareness
  • Cloud administration planning

Information for students

  • An AMI is the machine template used to launch an instance; it does not replace instance sizing decisions.
  • Launch templates improve repeatability by storing launch parameters once and reusing them.
  • CloudFormation is valuable because it turns infrastructure setup into a versionable and repeatable definition.
